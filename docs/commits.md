# Commit Message System
<!-- concept:def commit_conventions -->

- `init`  
  Initial project setup / first commit.  
  Example: `init: repo layer & types`
- `feat`  
  New feature implementation (usually fullstack).  
  Example: `feat: user sign up`
- `component`  
  New reusable UI component.  
  Example: `component: ProductBadges`
- `hooks`  
  Custom hooks or hook-related logic.  
  Example: `hooks: useUpdateProduct`
- `api`  
  API routes, services, controllers, backend logic.  
  Example: `api: getOrderById service`
- `style`  
  Styling, UI adjustments, design changes.  
  Example: `style: improve navbar spacing and colors`
- `layout`  
  Layout structure, page scaffolding, layout system.  
  Example: `layout: order page`
- `structure`  
  Folder structure, architecture, reorganization.  
  Example: `structure: move tanstack-query to /lib`
- `chore`  
  Maintenance, adjustments cleanup, config, tooling.  
  Example: `chore: update dependencies`
- `docs`  
  Documentation updates.  
  Example: `docs: update commit system`
- `fix`  
  Syntax, bug, or format fix.  
  Example: `fix: allow disabled on accept product button`
- `ref`  
  Refactors and technical debt cleanup.  
  Example: `ref: consolidate reusable cva variants across product UI`

---

## Commit Format Rules

- Always use lowercase titles
- Always use `:` after the type
- Keep messages concise and action-based
